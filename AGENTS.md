# 開発キットの規約

## 開発の流れ

1. `/grill-lite` で要件を短時間で整理し、仮定と未解決事項を明示する。
2. `/to-spec` で仕様を作成し、GitHub Issue に公開する。
3. `/to-tickets` で仕様を検証可能な縦方向のチケットに分解する。
4. `/implement` でチケットを実装する。合意済みの公開境界では `/tdd` を使う。
5. `/code-review` で、標準適合と仕様適合を並列サブエージェントで確認する。

## 課題管理

仕様と実装チケットは GitHub Issues で管理する。詳細は `docs/agents/issue-tracker.md` を参照する。

## ローカル成果物

ハーネスの実行中に生成する一時的な仕様、ローカルチケット、テスト出力、スクリーンショット、ログ、ビルド出力は `.scratch/<機能名>/` に置く。`.scratch/` は Git 管理しない。

テストコードやアプリケーションコードなど、再現に必要なソースは成果物ではないため Git 管理する。依存関係、カバレッジ、ブラウザテストレポートなどの再生成可能な出力はルートの `.gitignore` に従い Git 管理しない。

## ドメイン文書

用語はルートの `CONTEXT.md`、重要な判断は `docs/adr/` に日本語で記録する。既存の用語と ADR に矛盾する変更は、理由を明示してから行う。

## レビュー

コードレビューは固定点からの差分を対象にする。標準レビューと仕様レビューは、互いの判断に影響しないよう別々のサブエージェントで並列実行する。
