# チケットへ

計画、仕様、または会話を **チケット** のセットに分割します。つまり、トレーサー弾丸の垂直スライスであり、それぞれが**ブロック**するチケットを宣言します。

課題管理とトリアージラベルは `AGENTS.md`、`docs/agents/issue-tracker.md`、`docs/agents/triage-labels.md` に従います。これらがない場合は、GitHub Issue に公開する前に利用者へ運用方法を確認してください。

＃＃ プロセス

### 1. コンテキストを収集する

すでに会話のコンテキストにあるものから作業します。ユーザーが参照 (仕様パス、問題番号、または URL) を引数として渡す場合、それをフェッチし、その完全な本文とコメントを読み取ります。

### 2. コードベースを探索する (オプション)

まだコードベースを調べていない場合は、コードの現在の状態を理解するために調べてください。チケットのタイトルと説明には、プロジェクトのドメイン用語集の語彙を使用し、扱っている領域の ADR を尊重する必要があります。

実装を容易にするために、コードをプレファクタリングする機会を探してください。 「変化を簡単にしてから、簡単な変化をする。」

### 3. 垂直スライスをドラフトする

作業を **トレーサー弾丸** チケットに分割します。

<vertical-slice-rules>

- 各スライスは、すべてのレイヤー (スキーマ、API、UI、テスト) を通る狭いながらも完全なパスをカットします。1 つのレイヤーの水平スライスではなく、垂直にカットされます。
- 完成したスライスは、それ自体でデモ可能または検証可能です
- 各スライスは、単一の新しいコンテキスト ウィンドウに収まるようにサイズ設定されます。
- 事前ファクタリングは最初に行う必要があります

</vertical-slice-rules>

各チケットに **ブロッキング エッジ**、つまり開始する前に完了する必要がある他のチケットを指定します。ブロッカーのないチケットはすぐに開始できます。

**ワイド リファクタリングは垂直スライスの例外です。** **ワイド リファクタリング**は、列の名前を変更し、共有シンボルを再入力する 1 つの機械的な変更です。その **爆発半径**はコードベース全体に広がるため、1 回の編集で一度に数千の呼び出しサイトが破壊され、垂直スライスはグリーンに到達できません。無理に曳光弾に押し込まないでください。 **expand–contract** として順序付けします。まず展開します。何も壊れないように、古いフォームの横に新しいフォームを追加します。次に、ブラスト半径 (パッケージごと、ディレクトリごと) に応じたサイズのバッチで呼び出しサイトを移行します。各バッチは、拡張によってブロックされる独自のチケットを持ち、古い形式がまだ存在するため、バッチ間で CI グリーンを維持します。最後に契約します。すべての移行バッチによってブロックされたチケットで、呼び出し元が残っていない場合は、古いフォームを削除します。バッチだけでグリーンを維持できない場合は、シーケンスを維持しますが、最終的な統合と検証のチケットをすべてブロックする統合ブランチを共有させます。グリーンはそこでのみ約束されます。

### 4. ユーザーにクイズを出します

提案された内訳を番号付きリストとして提示します。各チケットについて、以下を表示します。

- **タイトル**: 短い説明的な名前
- **ブロックされている**: 他のチケット (存在する場合) が最初に完了する必要があるチケット
- **提供するもの**: このチケットが機能させるエンドツーエンドの動作

ユーザーに次のように尋ねます。

- 粒度は適切ですか？ (粗すぎる/細かすぎる)
- ブロッキング エッジは正しいですか。各チケットは、それを実際にゲートするチケットのみに依存していますか?
- チケットを統合またはさらに分割する必要がありますか?

ユーザーが内訳を承認するまで繰り返します。

### 5. 設定されたトラッカーにチケットを公開します。

承認されたチケットを公開します。公開方法は `docs/agents/issue-tracker.md` の設定に従います。GitHub Issue とローカルファイルではチケット内容は同じで、ブロッキングエッジの表現だけが異なります。

- **ローカル ファイル** → チケットごとに 1 つのファイルを `.scratch/<feature-slug>/issues/<NN>-<slug>.md` の下に書き込みます。`01` から依存関係の順序で番号が付けられます (ブロッカーが最初)。各ファイルの「ブロック対象」には、そのファイルが依存する番号/タイトルがリストされます。以下のチケットごとのファイル テンプレートを使用します。ファイルごとに 1 つのチケットを使用し、単一の結合ファイルを使用しないでください。
- **実際の問題トラッカー (GitHub、Linear など)** → チケットごとに 1 つの問題を依存関係の順序 (ブロッカーが最初) で公開し、各チケットのブロッキング エッジが実際の識別子を参照できるようにします。プラットフォームにネイティブのブロッキング/サブ問題関係がある場合は、それを使用します。それ以外の場合は、各チケットの「ブロック対象」をブロックする問題に設定します。特に指示がない限り、`ready-for-agent` トリアージ ラベルを適用します。チケットはエージェントが取得できるように構成されています。

**フロンティア** に取り組む: ブロッカーがすべて完了したチケット。上から下までを意味する純粋に直線的なチェーンの場合。

親問題を閉じたり変更したりしないでください。

**ブロックされている理由:** このチケットをゲートするチケットの番号/タイトル、または「なし - すぐに開始できます」。

**ステータス:** エージェントの準備完了

- [ ] 合格基準 2

## 何を構築するか

このチケットは、レイヤーごとの実装ではなく、ユーザーの観点から見たエンドツーエンドの動作を機能させます。

## 受け入れ基準

- [ ] 基準 1
- [ ] 基準 2

## によってブロックされました

- 各ブロッキング チケットへの参照、または「なし - すぐに開始できます」。

</issue-template>

どちらの形式でも、特定のファイル パスやコード スニペットは避けてください。これらはすぐに古くなってしまいます。例外: プロトタイプが、散文よりも正確に意思決定をエンコードするスニペット (ステート マシン、リデューサー、スキーマ、型の形状) を生成した場合は、それをインライン化し、それがプロトタイプからのものであることを簡単にメモします。決定事項が多い部分をトリミングします。実際に動作するデモではなく、重要な部分だけを切り取ります。

`/implement` を使用してフロンティアを一度に 1 枚ずつ処理し、チケット間のコンテキストをクリアします。
