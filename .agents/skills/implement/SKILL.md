ユーザーが仕様書またはチケットに記述した作業を実装します。

主成果物が `Webアプリ` の `Project Delivery` では、実装前にアプリの配置先が `apps/<kebab-case-app-name>/` であることを確認する。ハーネスルートにアプリ固有の `src/`、`test/`、`package.json` を作成してはならない。既存アプリが誤った場所にある場合は、新しい機能を追加する前に移動チケットを作成して配置を正す。

可能な場合は、事前に合意された継ぎ目で /tdd を使用してください。

合意された継ぎ目がある場合でも、実装を始める前に `/tdd` のテスト計画を作成する。会話または仕様に `Execution Mode: Uninterrupted` がある場合、テスト計画は記録・通知のみとし、承認待ちにせず実装へ進む。そうでない場合は承認を得る。継ぎ目を1つに絞ることは、テストケースを1本に絞ることではない。計画した振る舞いごとに Red → 最小実装 → Green を完了し、`quality/tdd-log.md` に証跡を記録してから次の振る舞いへ進む。複数の振る舞いを確認する最終スモークテストは、個別テストの代わりにしてはならない。

`Execution Mode: Uninterrupted` では、各チケットを完了するまで連続して実施する。レビューでP1またはP2の指摘が見つかった場合は、同じチケット内で修正 → 影響する個別テストと完全テストスイートの再実行 → 独立した再レビューを繰り返す。未解決のP1またはP2が1件でもある間は、チケットを完了にせず、次のチケットへの着手、完了報告、コミットをしてはならない。安全・権限上の判断が必要で、合理的な修正方針を選べない場合だけユーザーへエスカレーションする。

型チェックを定期的に実行し、単一のテスト ファイルを定期的に実行し、最後に完全なテスト スイートを 1 回実行します。

試作を作る場合、そのアプリコードとテストコードは `.scratch/<機能名>/prototype/` に置く。実行時データは `.scratch/<機能名>/data/`、テストの生データやスクリーンショットは `.scratch/<機能名>/test-results/` に保存する。これらを別の機能ディレクトリへ分散させない。

テストを実行するたびに、対象機能の `.scratch/<機能名>/quality/test-summary.md` を更新する。機能名は入力チケットまたは仕様書のパスから決める。各実行について、実行日時、コマンド、対象、成功/失敗、失敗時の証跡パスを記録する。最終結果は「合格」「不合格」「未実行」のいずれかを明記し、過去の実行結果を最新結果として扱わない。テスト計画にある各振る舞いとその最終判定を表で対応付け、未実装・未実行・不合格が1件でもあれば、全体を合格と報告してはならない。完全なテストスイートの前には、既存の `test-results/` を `.scratch/<機能名>/test-results/history/<実行日時>/` に退避し、古い失敗証跡と最新結果を混同しない。

完了したら、/code-review を使用して作業をレビューします。

コードレビュー後は、`.scratch/<機能名>/quality/code-review.md` が作成されていることを確認し、テスト要約から参照できるようにする。

コミット前に `test-plan.md`、`tdd-log.md`、`test-summary.md` と入力チケットを照合する。計画済みの振る舞いにRed/Greenの記録または正当な未実施理由がない場合、入力チケットの合格基準に未チェック項目がある場合、またはテスト結果とチケット状態が矛盾する場合は、実装完了・テスト合格・コミット可能とは扱わない。全ての合格基準が対応する個別テストで合格した後にだけ、チケットのチェックボックスを `[x]`、ステータスを `完了` に更新する。テストが未実行・不合格・未実装の項目は必ず未チェックのままとし、最終報告で明示する。

コードレビューのP1/P2も同じ完了ゲートに含める。`quality/code-review.md` にレビュー固定点、再レビュー回数、最終P1数、最終P2数、各指摘の解消テストを記録する。最終P1数または最終P2数が0以外なら、チケット状態を `完了` にしてはならない。

作業を現在のブランチにコミットします。
