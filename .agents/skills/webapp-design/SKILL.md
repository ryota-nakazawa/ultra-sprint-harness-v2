# Webアプリ設計

実装コードや詳細なコンポーネント仕様は作らない。利用者が実装前に確認できる `Design Brief` を会話内に出力する。UIイメージ確認が必要と判断した場合だけ、`Design Brief` の後に画像生成で主要画面または主導線の方向性画像を作る。入力にすでに含まれる要件を優先し、画面の重要な目的、最初の導線、最低限必要な画面が不明な場合だけ質問する。

## 前提を確認する

`grill-lite` の結果を読む。主成果物が `Webアプリ` でない場合は、このスキルを開始せず、`Artifact Routing` の `Next flow` に従う。

次を確認する。

- アプリが一言で解決する目的
- 利用者が最初に見るものと最重要操作
- 最低限の画面と、完了までの主導線
- 既存ブランド、デザインシステム、参考UIの指定
- UIの第一印象が受け入れ判断へ影響するか

## デザイン方針

既存ブランドや異なるテーマが明示されていない場合、現行 NTT DATA の公式サイトおよび現行CIを参照した青系テーマをデフォルトにする。白または明るいニュートラルを基調にし、青は主操作、状態、重要情報の識別に使う。ロゴ、商標アセット、厳密な公式カラー値は生成・同梱しない。

次のUI規範を必ず `Design Brief` に含める。

- 主操作は画面ごとに1つを基本とし、情報の重要度をタイポグラフィと余白で明確にする。
- 余白と読みやすさを優先し、装飾目的だけのカード、枠線、バッジを増やさない。
- 状態を色だけで表さない。十分なコントラスト、キーボード操作、意味的な見出し構造を保つ。
- UIの視覚品質が主な価値なら、`frontend-skill` を利用する。

## 画面と遷移

ユーザーの主目的、判断、入力・確認・完了などの状態、共有可能なURLまたは戻る操作が変わる箇所で画面を分ける。たとえば一覧、詳細、確認、完了は別画面にできる。

同じ目的の補助操作、短い選択、軽い編集は、独立ページにせずモーダル、ドロワー、またはインライン操作を優先する。画面数を増やすこと自体を目的にしない。

複数画面では、画面一覧と主要遷移を出力する。遷移の理解が重要な場合は、Mermaidによる簡潔な画面遷移図を含める。

## UIイメージ確認

LP、ダッシュボード、顧客向け画面など、第一印象、画面密度、情報配置、ブランド感が受け入れ判断に影響する場合は、実装前に画像生成で主要画面または主導線の方向性を確認する。確認する観点は、情報の優先順位、画面密度、色・トーン、最初に操作したくなる導線に絞る。

要件から画面像が明確な軽微な内部ツールでは、画像確認を省略できる。画像は方向性の確認用であり、実装の唯一の仕様にはしない。

画像確認が必要な場合は、`Design Brief` を出力した後に画像生成を実行する。画像では情報の優先順位、画面密度、色・トーン、最初に操作したくなる導線を確認し、利用者が方向性を確認してから `/to-spec` へ進む。

## 成果物を出力する

質問への回答が得られたとき、または質問なしで進められると判断したとき、次の形式で出力する。

```markdown
# Design Brief

## Purpose

## Theme

既定テーマまたは明示的な上書きテーマと、その理由。

## UI Principles

## Primary Action

## Screens

各画面の目的と、必要ならモーダル、ドロワー、インライン操作。

## Primary Flow

## Transition Diagram

複数画面で遷移理解が重要な場合だけ Mermaid を出力する。

## Visual Direction Checkpoint

必要／不要、理由、必要なら確認する観点。視覚品質が主な価値なら `frontend-skill` の利用も記載する。

## Next Step

画像確認が必要な場合は画像生成で方向性を確認し、その後 `/to-spec` に渡す。不要な場合はそのまま `/to-spec` に渡す。
```

`Design Brief` が目的、テーマ、UI規範、主操作、最低限の画面、主導線、画像確認の要否を含み、画像確認が不要または必要な画像生成と利用者の方向性確認が完了しており、重大な未解決事項がなければ `/to-spec` に渡す。
