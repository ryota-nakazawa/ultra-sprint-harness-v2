`HEAD` とユーザーが指定した固定点の差分を 2 軸で確認します。

- **標準** — コードはこのリポジトリの文書化されたコーディング標準に準拠していますか?
- **仕様** — コードは元の問題/PRD/仕様を忠実に実装していますか?

両方の軸が **並列サブエージェント** として実行されるため、互いのコンテキストが汚染されず、このスキルによって結果が集約されます。

品質レポートの保存先は `AGENTS.md` のローカル成果物規約に従う。仕様書またはチケットが `.scratch/<機能名>/` にある場合、品質レポートは `.scratch/<機能名>/quality/code-review.md` に保存する。機能名を特定できない場合は、固定点の確認とあわせて利用者に保存先を確認する。

＃＃ プロセス

### 1. 固定点を固定する

ユーザーが言ったことはすべて、コミット SHA、ブランチ名、タグ、`main`、`HEAD~5` などの固定点です。ユーザーが指定していない場合は、それを要求してください。

diff コマンドを 1 回キャプチャします: `git diff <fixed-point>...HEAD` (3 つのドットなので、比較はマージベースに対して行われます)。 `git log <fixed-point>..HEAD --oneline` によるコミットのリストにも注目してください。

次に進む前に、固定小数点が解決され (`git rev-parse <fixed-point>`)、diff が空でないことを確認してください。不正な ref または空の差分は、2 つの並列サブエージェント内ではなく、ここで失敗するはずです。

### 2. 仕様のソースを特定する

元の仕様を次の順序で探します。

1. ユーザーが引数として渡したローカルの仕様書またはチケットパス。
2. ブランチ名または機能と一致する `.scratch/` の下の `spec.md` またはチケットファイル。
3. コミットメッセージに参照されるローカルパス。
4. 何も見つからない場合は、仕様がどこにあるかをユーザーに尋ねます。存在しないと答えた場合、**Spec** サブエージェントはスキップして、「利用可能なスペックがありません」と報告します。

### 3. 標準ソースを特定する

コードの記述方法を文書化するリポジトリ内のすべて (`CODING_STANDARDS.md` や `CONTRIBUTING.md` など)。

リポジトリが文書化するものに加えて、標準軸には常に以下の**スメル ベースライン**が含まれます。これは、リポジトリに何も文書化されていない場合でも適用される、ファウラー コードのスメルの固定セット (_Refactoring_、ch.3) です。 2 つのルールがそれをバインドします。

- **リポジトリはオーバーライドされます。** 文書化されたリポジトリ標準が常に優先されます。ベースラインがフラグを立てそうなものを支持する場合は、臭いを抑制します。
- **常に判断が求められます。** それぞれの臭いはラベル付けされたヒューリスティック (「機能羨望の可能性」) であり、重大な違反ではありません。また、ここでの他の標準と同様に、ツールによってすでに強制されているものはすべてスキップしてください。

それぞれの臭いには *それが何なのか* → *修正方法* が記載されています。それを diff と照合します。

- **謎の名前** — 名前からは何をするのか、何を保持するのかが明らかでない関数、変数、または型。 →名前を変更します。正直な名前がなければ、デザインは曖昧になります。
- **重複したコード** — 変更内の複数のハンクまたはファイルに同じロジック シェイプが表示されます。 → 共有シェイプを抽出し、両方から呼び出します。
- **Feature Envy** — 自身のオブジェクトよりも別のオブジェクトのデータにアクセスするメソッド。 → メソッドを羨望のデータに移動します。
- **データの塊** — 同じいくつかのフィールドまたはパラメータが一緒に移動し続けます (生成を希望するタイプ)。 →それらを1種類に束ねて渡します。
- **Primitive Obsession** — 独自のタイプに値するドメイン概念を表すプリミティブまたは文字列。 → コンセプトに独自の小さなタイプを与える。
- **繰り返しの切り替え** - 同じタイプの同じ `switch`/`if` カスケードが、変更全体で繰り返し発生します。 → ポリモーフィズムに置き換えるか、両方のサイトが共有する 1 つのマップに置き換えます。
- **ショットガン手術** — 1 つの論理的な変更により、差分内の多くのファイルに分散した編集が強制されます。 → 変更内容を 1 つのモジュールにまとめます。
- **発散的な変更** — 1 つのファイルまたはモジュールが、いくつかの無関係な理由で編集されます。 → 各モジュールが 1 つの理由で変更されるように分割します。
- **推測的な汎用性** — 仕様にはないニーズのために追加された抽象化、パラメーター、またはフック。 →削除します。本当に必要性が分かるまでインラインバックします。
- **メッセージ チェーン** — 呼び出し元が依存すべきではない長い `a.b().c().d()` ナビゲーション。 → 最初のオブジェクトの 1 つのメソッドの後ろにあるウォークを非表示にします。
- **Middle Man** — 主に単に委任するだけのクラスまたは関数。 →カットして、本当のターゲットを直接呼び出します。
- **拒否された遺贈** — 継承内容のほとんどを無視またはオーバーライドするサブクラスまたは実装者。 → 継承を削除し、合成を使用します。

仕様またはチケットにテスト対象の振る舞いがある場合は、`.scratch/<機能名>/quality/test-plan.md`、`tdd-log.md`、`test-summary.md` も読む。計画済みの振る舞いに個別テストまたは正当な未実施理由がないことは、標準軸の問題として報告する。最終スモークテストだけでは、この確認を満たさない。

### 4. 両方のサブエージェントを並行して生成します

2 つの `Agent` ツール呼び出しを使用して 1 つのメッセージを送信します。両方に `general-purpose` サブエージェントを使用します。

**標準サブエージェント プロンプト** — 以下が含まれます:

- 完全な diff コマンドとコミット リスト。
- ステップ 3 で見つけた標準ソース ファイルのリスト、**およびステップ 3 の匂いベースライン** が完全に貼り付けられています。サブエージェントはこれ以外にアクセスできません。
- 概要: 「レポート — 関連するファイル/ハンクごと — (a) 差分が文書化された標準に違反しているすべての場所: 標準 (ファイル + ルール) を引用します。(b) 見つけたベースラインの臭い: 名前を付けてハンクを引用します。重大な違反と判断の判断を区別します — 文書化された標準違反は難しい場合がありますが、ベースラインの臭いは常に判断の判断であり、文書化されたリポジトリ標準がベースラインをオーバーライドします。ツールによる強制はすべてスキップします。 400語ですよ。」

**サブエージェント プロンプトの指定** — 以下が含まれます。

- diff コマンドとコミット リスト。
- 仕様のパスまたはフェッチされたコンテンツ。
- 概要: 「レポート: (a) 仕様で要求された、欠落または部分的な要件。(b) 要求されなかった diff 内の動作 (スコープ クリープ)。(c) 実装されているように見えるが、実装が間違っているように見える要件。各結果について仕様行を引用します。400 語以内。」

仕様が欠落している場合は、仕様サブエージェントをスキップし、最終レポートでこれをメモしてください。

### 5. 集計

`## Standards` および `## Spec` という見出しの下にある 2 つのレポートをそのまま、または軽くクリーンアップして提示します。結果を結合したり、再ランク付けしたりしないでください**。2 つの軸は意図的に分離されています (_2 つの軸がある理由_を参照)。

1 行の要約で終了します。軸ごとの合計結果と、_各軸内の最悪の問題 (存在する場合)。さらに、レビュー全体の未解決P1数・未解決P2数を明記する。複数の軸にわたって単一の勝者を選択しないでください。これは、分離が防止するために存在する再ランキングです。

会話へ返す前に、集計内容を `.scratch/<機能名>/quality/code-review.md` に保存する。レポートには固定点、レビュー日時、差分コマンド、仕様書のパス、次のサブエージェント実行表を含める。

| 役割 | 実行方法 | 対象 | 状態 |
| --- | --- | --- | --- |
| Standards | 並列サブエージェント | 文書化された標準とコードスメル | 完了または失敗 |
| Spec | 並列サブエージェント | 仕様との適合性 | 完了、失敗、または仕様なし |

この表の後に `## Standards` と `## Spec` の完全な結果を記載する。末尾に `## Exit Gate` を置き、未解決P1数・未解決P2数・各指摘を解消するテスト・次の行動を記録する。レビューに使用したサブエージェントをこのファイルから判別できるようにする。

## なぜ 2 つの軸なのか

変更によって一方の軸が通過し、もう一方の軸が失敗する可能性があります。

- すべての標準に従っているが、間違ったものを実装しているコード → **標準は合格、仕様は不合格**
- 問題の内容を正確に実行するが、プロジェクトの規約に違反するコード → **仕様は合格、標準は不合格**

それらを個別にレポートすると、一方の軸が他方の軸をマスクすることがなくなります。
